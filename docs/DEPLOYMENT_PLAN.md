# Deployment Plan: Qwen2.5-VL-7B REST Service

## 1) Objective
Deploy a REST API service that accepts an image or video file plus a text query, then returns a model response from `Qwen/Qwen2.5-VL-7B-Instruct`.

Requirement decisions applied:
- Single-machine deployment: Docker-first.
- Authentication: not required.
- Batch input: supported (`/v1/query/batch`) plus UI-side sequential coordination option.
- Video pipeline choice: `sampled_frames` and `full_temporal`.
- Production observability: request IDs, latency, GPU memory, runtime metrics endpoint.

## 2) High-Level Architecture
- API Layer:
  - `POST /v1/query` for single-file inference.
  - `POST /v1/query/batch` for multi-file inference.
  - `GET /health` and `GET /metrics`.
- Inference Layer: Transformers `Qwen2_5_VLForConditionalGeneration` + `AutoProcessor`.
- Modal Input Handling:
  - Image path: direct PIL processing.
  - Video path option A: frame sampling via OpenCV (`sampled_frames`).
  - Video path option B: full temporal video input via `qwen-vl-utils` (`full_temporal`).
- Test Interface:
  - Deterministic media assets in `tests/assets`.
  - Scripted REST tests in `tests/test_api_requests.py`.
  - Browser UI at `/` from `app/static/index.html`.
- Observability:
  - Middleware-level request logging with request ID, latency, and GPU memory stats.
  - Request ID returned via `X-Request-ID`.

## 3) Runtime Prerequisites
- OS: Windows/Linux.
- Python: 3.10+.
- Recommended GPU: NVIDIA with at least 16 GB VRAM.
- Disk: at least 20 GB free for model weights and runtime artifacts.

## 4) Deployment Steps
1. Build and run Docker container for single-machine setup:
  - CPU: `docker compose up --build vlm-api`
  - GPU: `docker compose --profile gpu up --build vlm-api-gpu`
2. Optional non-container run:
  - Create venv, install `requirements.txt`, run `uvicorn app.main:app --host 0.0.0.0 --port 8000`
3. Generate deterministic test media (for local testing scripts):
  - `python tests/generate_deterministic_media.py`
4. Validate:
   - Health check: `GET /health`
  - Runtime metrics: `GET /metrics`
   - Browser UI: `GET /`
  - Scripted test: `python tests/test_api_requests.py --run-batch`

## 5) API Contract
- Route: `POST /v1/query`
- Content-Type: `multipart/form-data`
- Fields:
  - `query` (string, required)
  - `file` (image/video file, required)
  - `max_video_frames` (int, optional, default `8`)
  - `video_pipeline` (`sampled_frames` or `full_temporal`, optional)
- Success Response:
  - `model`: model id
  - `query`: original query
  - `filename`: uploaded file name
  - `answer`: generated response text

- Route: `POST /v1/query/batch`
- Content-Type: `multipart/form-data`
- Fields:
  - `query` (string, required)
  - `files` (multiple image/video files, required)
  - `max_video_frames` (int, optional)
  - `video_pipeline` (`sampled_frames` or `full_temporal`, optional)
- Success Response:
  - `results`: per-file inference outputs
  - `errors`: per-file failures (if any)

## 6) Deterministic Testing
- Test assets generated by `tests/generate_deterministic_media.py`:
  - `tests/assets/deterministic_sample.png`
  - `tests/assets/deterministic_sample.mp4`
- REST verification script:
  - `tests/test_api_requests.py`

## 7) Known Constraints
- Initial request latency is high due to model download and load.
- CPU inference is possible but slow for 7B VL model.
- `full_temporal` has higher memory pressure and may be slower than `sampled_frames`.
- Batch endpoint executes files sequentially to protect stability; client-side sequential mode is provided in UI.

## 8) Next Hardening Items
- Add request rate limits and overload protection.
- Add persistent metrics sink (Prometheus/OpenTelemetry).
- Add queueing and worker pool for heavy concurrent loads.
- Add model warmup and autoscaling policy for non-single-machine environments.
